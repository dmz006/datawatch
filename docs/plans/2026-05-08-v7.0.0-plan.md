## 1. Vision

datawatch v7.0.0 is the major architectural shift from "Council Mode is a stub" to **real, federated, observable LLM workloads** managed by datawatch as first-class infrastructure:

- A **ComputeNode** is anywhere local LLM workloads run (a host, a GPU box, a cluster behind a load balancer, a containerized runtime, a remote-proxied datawatch peer). Each ComputeNode has identity, declared capacity, monitoring, RBAC, scheduling priority, costs, and maintenance windows.
- An **LLM registry** lets operators define `(name, kind, model, ordered_compute_node_failover_list)` once and have every datawatch consumer (Council, /api/ask, BL297 persona-wizard, automata spawn, agents, sessions) call them through a single dispatcher.
- **Real Council debates** stop emitting STUB strings. Each persona response is a real LLM call. By default, council runs as parallelized async sessions with SSE live updates streaming round-by-round results to the Automata UI, mobile app, and comm channels.
- **Each persona response is a session.** Operators can `datawatch attach council-<run>-<persona>` to watch any persona's reasoning stream in real time.
- **Persistent personas have layered memory** (persona-global → persona-in-project → project-shared → session-local) with explicit cross-session borrowing and operator-controlled promotion.

This is the foundation for: federated GPU pooling across operator infrastructure, cost-aware scheduling, auditable persona reasoning, persistent expert agents, and future per-persona prioritization/weighting.

## 4. Architecture overview

### 4.1 Concept layers

```
┌─────────────────────────────────────────────────────────────────────┐
│ Consumers                                                           │
│   Council orchestrator | /api/ask | BL297 persona wizard            │
│   Automata spawn       | Agent runtime | Session spawn              │
└──────────────────────────────┬──────────────────────────────────────┘
                               │ uses
                               ▼
┌─────────────────────────────────────────────────────────────────────┐
│ LLM registry  (cfg.llms[])                                          │
│   each LLM = name + kind + model + ordered ComputeNode failover     │
│   dispatcher routes calls per LLM definition                        │
└──────────────────────────────┬──────────────────────────────────────┘
                               │ runs on
                               ▼
┌─────────────────────────────────────────────────────────────────────┐
│ ComputeNode registry  (cfg.compute_nodes[])                         │
│   each Node = where workloads run + capacity + monitoring + RBAC    │
│   (local | ssh | docker | k8s | remote-proxy)                       │
└──────────────────────────────┬──────────────────────────────────────┘
                               │ observed by
                               ▼
┌─────────────────────────────────────────────────────────────────────┐
│ Monitoring stub                                                     │
│   datawatch agent on each Node reports CPU/mem/GPU/concurrency      │
│   (used for v1 declared-capacity validation; v2 dynamic scheduling) │
└─────────────────────────────────────────────────────────────────────┘
```

### 4.2 Council run lifecycle

```
operator → POST /api/council/run (proposal, mode=quick|debate, personas?)
       │
       ▼
   create Run row (id, status=running, mode, personas, rounds_total, MaxParallel)
       │
       ▼
   for each round:
     for each persona (concurrent up to MaxParallel):
       spawn datawatch session "council-<runid>-<persona>-r<n>"
         shell pipes prompt → chosen LLM (via LLM registry)
         streams response back to Run row + SSE channel
       on persona error → retry once → fall through to next host in LLM's
         failover list → if all hosts fail, mark persona response = "[name] error"
       persistent personas: load memory layers; write learnings to session-local
   after final round:
     synthesize call (also uses LLM registry)
   set status=completed | interrupted | cancelled
```

### 4.3 Memory layers

```
persona-global  ─┐  (per persona, all projects)         [explicit promote]
                 │
persona-in-project ┐  (per persona, current project)    [explicit promote]
                   │
project-shared  ───┐  (current datawatch episodic memory, cross-persona)
                   │                                    [explicit promote]
session-local  ────┘  (per council run, default write target)

Recall: walks top-down, returns merged result with layer attribution.
Borrow (cross-session, read-only): memory_recall(namespace="session-X")
Seed  (cross-session, copy):     memory_seed --from session-X --to session-Y --filter ...
Promote: memory_promote --memory <id> --to <layer>
```

### 4.4 Live updates

- **Automata UI / mobile app:** SSE on `/api/council/runs/{id}/events`. Events: `run_started`, `round_started`, `persona_responding`, `persona_response`, `persona_error`, `round_completed`, `synthesis_started`, `run_completed`, `run_cancelled`.
- **Comm channels:** push messages at run start, each round end, errors, synthesis complete (no per-persona spam).
- **Operator injection:** `POST /api/council/runs/{id}/personas/{persona}/inject` body `{instruction}` — appends to that persona's next-round prompt. Future, but architecture must reserve the endpoint shape.

## 9. Risk register

| Risk | Mitigation |
|------|------------|
| LLM-call latency makes UX feel broken | SSE live updates (S4) + per-call timeout + visible progress |
| Cost runaway on SaaS LLMs (claude, openai, gemini) | Per-LLM cost caps in S2; alert at 80% of budget; future SaaS-only "manual" failover |
| ComputeNode misconfiguration silently routes calls into a black hole | S2 dispatcher requires LLM `test` to pass before LLM is marked active; UI shows red-dot status |
| Memory promotion pollutes higher layers with junk | Operator-only promote (default); persona-driven promotion only for personas with explicit prompt instruction; v2: LLM grader gate |
| Cross-session memory borrow creates information leaks (web project leaks to marketing) | Read-only by design; explicit operator seed action; no automatic merge |
