## Decision Log

| # | Decision | Rationale |
|---|---|---|
| D1 | Resources = pull context only; no subscriptions in v7.1.0 | Claude Code has no built-in behavior on resource update notifications; subscriptions go in v7.2.0 after we observe usage |
| D2 | Resource URI scheme: `datawatch://` | Clean namespace; channel bridge already forwards all MCP calls via REST proxy |
| D3 | Sampling included in v7.1.0, alongside elicitation | Both are daemon-initiated; share plumbing; build once |
| D4 | Elicitation alongside sampling but does NOT replace permission relay | Relay has proven semantics and deployed clients; replace in v7.2.0 |
| D5 | Prompts reduced to 10 focused slash commands | Sampling covers daemon-initiated flows; prompts cover operator-initiated |
| D6 | Plugin/skill manifest declares sampling triggers | Plugins can request Claude analysis as part of their lifecycle |
| D7 | Tool annotations added (readOnly/destructive) on all existing tools | Low effort; Claude Code uses annotations for permission UX |
| D8 | Council sampling: daemon includes Claude Code as deliberation member | Natural fit with Council architecture; wired in S3 |
| D9 | No subscriptions, no roots, no completions in v7.1.0 | Deferred to v7.2.0 after seeing how clients use the read surface |

#### Resource content format

Each handler returns `mcpsdk.TextResourceContents` with JSON content for structured data, or `mcpsdk.TextResourceContents` with Markdown for docs. MIME types:
- JSON resources: `application/json`
- Docs: `text/markdown`

#### Live handler pattern

Each live handler calls the same internal data sources the REST API uses, not the REST API itself (avoids loopback HTTP overhead):

```go
func (s *MCPResourceServer) handleSessionsResource(ctx context.Context, req mcpsdk.ReadResourceRequest) ([]mcpsdk.ResourceContents, error) {
    sessions := s.sessionStore.ListActive()
    data, _ := json.Marshal(sessions)
    return []mcpsdk.ResourceContents{
        mcpsdk.TextResourceContents{URI: req.Params.URI, MIMEType: "application/json", Text: string(data)},
    }, nil
}
```

#### Task 3.14тАУ3.15 тАФ Channel bridge: sampling + elicitation

**File**: `cmd/datawatch-channel/main.go`

At startup after `bridge.mcpSrv.EnableSampling()`:

The channel bridge is the *client* end for sampling тАФ it's Claude Code's proxy. The daemon sends `sampling/createMessage` via the MCP connection; the channel bridge receives it and forwards to Claude Code's stdio session.

func (s *MCPPromptServer) handleAnalyzeSession(ctx context.Context, req mcpsdk.GetPromptRequest) (*mcpsdk.GetPromptResult, error) {
    id := req.Params.Arguments["session_id"]
    if id == "" { id = s.activeSession() }
    sessionData, _ := s.readResource(ctx, "datawatch://sessions/"+id)
    historyData, _ := s.readResource(ctx, "datawatch://sessions/"+id+"/history")

    return &mcpsdk.GetPromptResult{

        Messages: []mcpsdk.PromptMessage{

            }},
        },
    }, nil
}

## Architecture Notes

### How sampling flows

```
daemon (trigger fires)
  тЖТ SamplingDispatcher.Sample(req)
    тЖТ srv.RequestSampling(ctx, CreateMessageRequest)
      тЖТ active stdio session (Claude Code)
        тЖТ Claude processes messages
        тЖТ returns CreateMessageResult
      тЖР result
    тЖР SamplingResult
  тЖР caller acts on result (log / route to comm / update automaton state)
```

If no Claude Code session is connected: `ErrSamplingNotSupported` returned. Caller falls back to direct Ollama/API call. Sampling is always additive, never blocking.

### How elicitation flows

```
daemon (decision point reached)
  тЖТ ElicitationDispatcher.Elicit(req)
    тЖТ srv.RequestElicitation(ctx, ElicitationRequest{Schema: "approval", ...})
      тЖТ Claude Code presents structured input to operator
        тЖТ operator responds
      тЖР ElicitationResult{action: "approve", note: "..."}
    тЖР ElicitationResult
  тЖР automaton/plugin continues with operator's decision
```

If no session or client doesn't support elicitation: falls back to channel notification (existing relay path).
