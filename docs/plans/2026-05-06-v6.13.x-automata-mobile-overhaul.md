## Questions needing operator input

### Q1 — Wizard concept (A4)

Operator's confusion: *"i thought we were doing a wizard where 'type' was inferred? Not sure why 'inferred' is on the launch automaton if we are selecting type."*

Three paths to resolve:

- **(a) Pure inferred** — drop the Type chip selector entirely. Show the inferred type as a small label below the intent textarea ("Detected: software"). Operator can override only via Advanced.
- **(b) Pure selected** — drop the "Inferred" header. Operator picks type from a chip row at the top of the form.
- **(c) Inferred-with-override** — show "Detected: software ✏️" pill below the intent textarea; tap to expand a chip row of all types for override. The "Inferred" section header goes away; type is part of the intent context.

**My recommendation: (c)** — the wizard already auto-infers from intent text; making it visible + tap-to-override matches the spirit + cleans up the UI. (a) is too magic; (b) loses the inference value.

### Q2 — Project Profile Skills field (A9)

Operator: *"is that in the agent settings? I look at profile settings in settings/agent/profiles and there is no skills option or listing, should there be?"*

**Implicit answer: yes.** Project Profiles need a Skills multi-select field; the wizard's "available in agent profile" copy then makes sense (it points at where the operator can permanently configure).

Confirming: add Skills field to Project Profile editor. **Going with yes.**

### Q3 — Rules Check tab (C10) + Run Scan placement (C11) + Conditional tab visibility (C12)

Operator: *"shouldn't there also be a tab for rules check (if enabled) with similar cards showing the rules check tests?"* + *"should scan page and rules check pages have 'run scan' or should those be action buttons at top of automata since if they are enabled it should be part of the workflow"* + *"scans and rules check should be visible only if they are enabled"*

Three sub-decisions:

- **C10 — Rules Check tab**: yes when `guardrails: rules` is in the spec. **Going with yes.**
- **C11 — Run Scan placement**: top action button, NOT inside the Scan tab. The Scan tab becomes a history/results view; the "Run Scan" verb lives with the other top-level actions (Approve / Run / Cancel). **Going with top action button.**
- **C12 — Conditional visibility**: hide Scan tab when `guardrails: security` not in spec; hide Rules Check tab when `guardrails: rules` not in spec. **Going with conditional.**

### Q4 — Action button placement on automata cards in the LIST view (C1)

Operator: *"there are now buttons, that should be on all automata i noticed not all had all buttons not sure why"*.

Two interpretations:

- **(a) List-row action buttons** — each row in the Automata list shows the action set inline. **High visual noise on mobile.**
- **(b) Detail-view action buttons** — every automaton in detail view should have the full action set; that's where C1 lives.

**My read: (b).** The list view is for navigation; detail view is for action. (a) would clutter the list without payoff. Let me know if you meant (a).

## Implementation plan

Sequenced for batched landing as **v6.13.1** (PWA-only patch) and
**v6.13.2** if the operator wants the C-group treated as separate
(it's the bigger chunk).

### Phase 1 — Launch Automation wizard (Group A)

Targeting v6.13.1.

- A1 — fix Start-from-template strip CSS (proper flex row, single-line, no wrap on mobile).
- A2 — add `🎤` mic button to large text inputs (intent textarea + others); reuse the existing voice-transcribe code path.
- A3 — intent textarea: `width: 100%`; min `rows: 4`; auto-grow.
- A4 — implement wizard model (c) per Q1: drop "Inferred" header; show "Detected: software ✏️" pill below intent; tap = expand chip row.
- A5–A8 — global tightening pass: `wizard-section { padding: 6px 10px }` (was 12px+), checkbox row gap 4px, type chips fit in one mobile row (`flex-wrap: nowrap; overflow-x: auto`), labels 11px → 12px for readability.
- A9 — add `Skills` multi-select to Project Profile editor; wizard copy updated to "Available in this profile (Settings → Agents → Project Profiles)".

### Phase 2 — Multi-select bug + List-view fix (Group B)

Targeting v6.13.1.

- B1 — investigate "command not known" error when running multi-select actions. Probably a router fall-through after the v6.12.4 state-aware refactor; needs reproduction + handler fix.

### Phase 3 — Individual automata view (Group C)

Targeting v6.13.2 (separate cut so the operator can review the
wizard fixes first; if they want both in one cut, fold Phase 3 into
v6.13.1).

- C1 — audit which states drop which action buttons; ensure every reachable state has the full action set.
- C2 — remove the "…" / Plan dropdown.
- C3 — split toolbar into two rows: **Actions** (Approve / Run / Cancel / Run Scan / Run Rules Check / Archive / Delete) and **Edit** (Edit Spec / Settings / Request Revision / Clone to Template).
- C4 — convert the tab strip from `<button>` to the same `output-tab` class used in session detail (parity with the in-session tab strip).
- C5 — settings modal: apply the same global tightening pass as A5-A8.
- C6 — Stories tab cards: redesign per session-list / automata-list card pattern. Per-story icons unified (Edit / Files / Approve / Reject); pencil icon explained inline.
- C7 — Stories cards: card-based with proper mobile padding.
- C8 — Decisions tab: render each decision as a card (timestamp + actor + action + reason); expandable for raw `details` payload.
- C9 — Scan tab: card-based result list; clear mobile action buttons; "Re-run" per-result.
- C10 — Add Rules Check tab when `rules` guardrail is enabled.
- C11 — Move Run Scan to the top Action row (gone from inside the Scan tab).
- C12 — Conditional tab visibility based on enabled guardrails.

### Phase 4 — Edit Profile header (Group D)

Targeting v6.13.1 (small).

- D1 — strip "Edit project profile: XXX" prefix from the modal title; render the profile name as the heading. Same for cluster profile editor.
