**Known-set ordered badge:** same as known-set but badges are draggable (for fallback chain ‚ÄĒ order matters). `type: 'known-ordered', source: '/api/...'`.

Replaces the v5.26.8 `csvExpandButtonHTML` pattern. The existing `csvExpandButtonHTML` function remains for any remaining non-settings CSV usage but is no longer used in settings forms.

## v8.2.0 ‚ÄĒ BL328: PRD decompose async (SSE + polling)

**Filed from GitHub #77. Operator-filed 2026-05-19.**  
**Version:** v8.2.0  
**Sprint prefix:** T40b  
**Test story range:** TS-645 ‚Äď TS-658  
**Unblocks:** Android TS-232, TS-233, TS-234, TS-236, TS-237, TS-239, TS-240, TS-241

### Problem

`POST /api/autonomous/prds/{id}/decompose` calls the LLM synchronously, blocking the HTTP connection. For local Ollama models on non-trivial PRDs this exceeds the 30‚Äď60s client timeout. All Android T13 autonomy lifecycle stories are blocked.

### Architecture

```
POST /api/autonomous/prds/{id}/decompose
  ‚Üí 202 Accepted
    { "task_id": "<id>", "stream_url": "/api/autonomous/prds/{id}/decompose/stream" }

Background goroutine:
  - Calls LLM decompose
  - Writes each story to DB as it's generated (persists across daemon restart)
  - Updates decompose_state: pending ‚Üí in_progress ‚Üí complete | error

GET /api/autonomous/prds/{id}/decompose/stream   ‚Üź SSE (primary client path)
  text/event-stream
  data: {"type":"story","title":"...","description":"..."}
  data: {"type":"progress","done":3,"total":7}
  data: {"type":"complete","story_count":7}
  data: {"type":"error","message":"..."}

GET /api/autonomous/prds/{id}/decompose/status   ‚Üź polling fallback (API utility)
  { "status": "pending|in_progress|complete|error",
    "progress": 3, "total": 7, "stories": [...] }
```

### Reconnect / backoff behavior (both clients)

**SSE reconnect:** on disconnect ‚Üí exponential backoff starting 1s, doubling, capping at 30s. On reconnect, re-open stream from `Last-Event-ID` header (server sends `id:` field per event so client can resume). Low-power: pause on document hidden (PWA) / ConnectivityManager network loss (Android).

**Fallback to polling:** when SSE reconnect backoff exceeds 15s, switch to polling status endpoint at 5s intervals. Resume SSE when network recovers.

**VPN / Tailscale (Android):** `ConnectivityManager.NetworkCallback` ‚ÄĒ pause all activity immediately on network loss; resume on `onAvailable()`. Best-effort: trigger Tailscale reconnect intent once after 10s of network loss, then wait. Do not retry the VPN trigger.

**PWA:** `document.addEventListener('visibilitychange')` ‚ÄĒ pause SSE when hidden, resume on focus. `window.addEventListener('online')` ‚ÄĒ resume SSE on browser online event.

**Server-side restart safety:** background goroutine writes each story to DB before emitting SSE event. On daemon restart, the status endpoint reflects the last persisted state. Client can re-open the stream; server re-emits events for already-generated stories from DB.

## v8.3.0 ‚ÄĒ BL331: Channel-Address Federation via Comms

**Operator-filed 2026-05-19.**  
**Version:** v8.3.0  
**Sprint prefix:** T41  
**Test story range:** TS-683 ‚Äď TS-720 (estimated)

### Background

The channel bridge (Signal/Matrix/Discord/Slack/etc.) was designed for operator communication. External trial users (Cairn-works) revealed a second use case: a federated AI instance using the channel bridge as its federation surface ‚ÄĒ not REST/MCP/CLI, but a chat channel. This is federation through comms.

This builds on BL316 (CBAC) by extending the channel bridge to be federation-peer-aware at the sender level.

### Design decisions (interview 2026-05-19)

- **Sender auth:** `channel_identity[]` on federation peer config. Federation must be pre-established by operator or AI via MCP/API. Each identity entry has `type` (matrix_dm, matrix_room, slack_dm, slack_room, signal, discord_dm, discord_server) and `value`. Future: key-based auth for initial bootstrap (deferred).
- **1:1 vs group rooms:** 1:1 DM channels ‚ÄĒ no address token prefix needed; whole message is the command. Group rooms ‚ÄĒ address token prefix required. The `channel_identity` type discriminates.
- **Command format:** Structured commands with catch-all. Unknown command routes to: `run <message>` ‚Üí starts Automaton with message body as intent. Structured first ‚ÄĒ AI peers should use a defined command surface; easier to debug.
- **Routing config:** Global default (all comms-channel peers) + per-peer override.
- **Ownership:** `owner_peer` field on Session + PRD. Comms-channel agents own what they create. `own-only` scope is the default; `view_all` grantable per peer.

### Architecture

```
Incoming channel message (Matrix, Slack, Signal, Discord, etc.)
  ‚Üď
Channel bridge receives message
  ‚Üď
Look up sender in federation peer channel_identity list
  ‚Üď (not found ‚Üí treat as operator message, existing behavior)
  ‚Üď (found ‚Üí federated comms path)
Apply CBAC: peer has comms-channel-agent capabilities?
  ‚Üď (no ‚Üí 403, reply with error to channel)
  ‚Üď (yes ‚Üí route to configured handler)
Global routing config + per-peer override:
  handler=automata ‚Üí spawn Automaton (message body = intent, owner_peer=peer_name)
  handler=plugin   ‚Üí run configured Plugin handler
  handler=mailbox  ‚Üí store in Discussion Scope (BL332, deferred to v8.4)
  ‚Üď
On completion: post response back to originating channel (or 1:1 reply)
```

### New capability group: `comms-channel-agent` (14th built-in)

| Capability | Scope |
|---|---|
| `autonomous:list/read/write/run` | own-only by default |
| `sessions:list/read` | own-only by default |
| `dashboard:read` | global read |
| `alerts:list/read` | own sessions only |
| **Does NOT include** | `config:*`, `federation:write`, `secrets:*`, `sessions:kill` (others'), `sessions:input` (others') |

### Config schema extension

```yaml
federation_peers:
  - name: cairn-terminal
    url: https://cairn.substrate:8080
    token: tok-cairn
    capabilities: ["comms-channel-agent"]
    channel_identities:
      - type: matrix_dm    # 1:1 DM ‚ÄĒ no address token needed
        value: "@cairn:matrix.org"
      - type: matrix_room  # Group room ‚ÄĒ address token required
        value: "@cairn:matrix.org"
    address_tokens: ["cairn-terminal:", "@cairn-terminal", "for cairn:"]
    channel_handler: automata   # per-peer override; falls back to global
    view_all: false             # own-only scope (default)

comms_channel:
  default_handler: automata    # global default for all comms-channel-agent peers
  catch_all_template: ""       # template name for catch-all; empty = inline PRD
```

### Session + PRD schema extension

```go
type Session struct {
    // existing fields ...
    OwnerPeer string `json:"owner_peer,omitempty"` // federation peer that created this
}

type PRD struct {
    // existing fields ...
    OwnerPeer string `json:"owner_peer,omitempty"`
}
```

CBAC filter for comms-channel-agent list operations: `WHERE owner_peer = <peer_name> OR view_all = true`.

## v8.3.0 ‚ÄĒ BL333: Federated File Service

**Operator-filed 2026-05-19.**  
**Version:** v8.3.0  
**Sprint prefix:** T43  
**Test story range:** TS-701 ‚Äď TS-725 (estimated)

### Background

Workspace sync (#70) is solvable via NFS/rsync/git as operator-configured Plugin/Skill patterns (community templates filed). But a first-class federated file service in the daemon:
1. Enables peer-to-peer artifact transfer using existing federation auth
2. Unlocks PWA file upload (long-missing)
3. Unlocks mobile file picker ‚Üí server
4. Provides a clean transport for workspace sync without requiring operator-side rsync config

Community plugin templates (NFS, git, rsync) filed to `dmz006/datawatch-community` as the lightweight path.

### Storage isolation model

**Critical security requirement (operator-confirmed 2026-05-19):** A flat global root with CBAC is insufficient ‚ÄĒ a peer with `files:read` would see all files from all peers and all discussions. The same isolation principle from BL332 applies: **access boundary = storage boundary**. No peer can enumerate or infer the existence of another peer's files or discussion artifacts.

**Directory layout under `file_service.root`:**

```
<file_service.root>/
  local/                          # operator/self files ‚ÄĒ never visible to peers
  peers/
    <peer_name>/                  # isolated per-peer directory
      ...                         # peer can only see their own subtree
  discussions/
    <discussion_id>/              # shared between discussion participants only
      ...                         # non-participants cannot list, read, or infer this dir exists
```

**Access rules (enforced at API layer, not application layer):**

| Requester | Can access |
|---|---|
| Operator (admin token) | `local/`, all `peers/*/`, all `discussions/*/` |
| Federation peer "cairn" | `peers/cairn/` only + `discussions/<id>/` where "cairn" is a participant |
| comms-channel-agent peer | `peers/<own_name>/` + own discussion directories |
| Peer with no discussions | `peers/<own_name>/` only ‚ÄĒ cannot enumerate `discussions/` |

**No enumeration of other directories.** `GET /api/files` with a peer token returns only their visible subtree. A `404` (not `403`) is returned for paths outside their boundary ‚ÄĒ no information about whether the path exists is leaked.

### Config

```yaml
file_service:
  enabled: true
  root: ~/.datawatch/files        # absolute or ~ path; required
  max_file_size_mb: 100           # default
  allowed_extensions: []          # empty = all allowed
  per_peer_quota_mb: 500          # optional per-federation-peer quota
```

### Capabilities

```
New capabilities (added to federation capability set):
  files:list, files:read, files:write, files:delete

New built-in group additions:
  comms-channel-agent: + files:read, files:write (own peer directory + own discussions)
  read-only:           + files:list, files:read  (own peer directory only)
  full-control:        + all files:* caps
```

### Endpoints

```
# All paths are relative to the caller's visible root (peers/<name>/ or discussions/<id>/)
GET  /api/files                    list files in caller's visible root
GET  /api/files?path=<subdir>      list subdirectory (must be within caller's boundary)
POST /api/files/upload             multipart upload; returns { path, size, sha256 }
GET  /api/files/download/{path}    download file (404 if outside caller's boundary)
DELETE /api/files/{path}           delete file (404 if outside caller's boundary)

# Discussion-scoped file access (requires BL332 discussion participation)
GET  /api/files/discussions/{discussion_id}             list discussion files
POST /api/files/discussions/{discussion_id}/upload      upload to discussion dir
GET  /api/files/discussions/{discussion_id}/{path}      download from discussion dir

# Peer-to-peer transfer (pushes to remote peer's own directory)
POST /api/files/transfer
  { "peer": "cairn", "src": "relative/path", "dst": "optional/remote/path" }
  ‚Üí 202 Accepted; proxied to peer's POST /api/files/upload via federation token
  ‚Üí file lands in peer's peers/<own_name>/ directory on their server

# Storage metadata (advertises this server's file service to federation peers)
GET  /api/federation/peers/{name}/storage
  ‚Üí { "quota_mb": 500, "used_mb": 42, "file_service_enabled": true }
  (does NOT reveal root path; requires files:list cap)
```

## v8.4.0 ‚ÄĒ BL332: Discussion Scopes (federated shared memory)

**Operator-filed 2026-05-19. Merges GH#68 + GH#69.**  
**Version:** v8.4.0  
**Sprint prefix:** T42 (3 sprints: T42a / T42b / T42c)  
**Test story range:** TS-719 ‚Äď TS-750 (estimated; expanded per sprint)

### Background

GH#68 (Channel‚ÜĒMailbox Relay) and GH#69 (Cross-substrate Mailbox Sync) are the same feature: a shared persistent memory scope between 2+ instances. "A discussion is just a shared memory between instances." Filed as one BL.

The existing Memory scope hierarchy (persona-global ‚Üí persona-in-project ‚Üí project-shared ‚Üí session-local) is hierarchical ‚ÄĒ parent/child. Discussion scopes are peer-equal. This requires new scope type and sync protocol, but shares the existing memory storage backend.

### Design decisions (interview 2026-05-19)

| Decision | Outcome |
|---|---|
| Conflict resolution | Surface + resolve (not silent). Store both conflicting entries with markers. Resolution via API/MCP. |
| Sync direction | Push-on-write. No polling. Delivery queue for offline peers. |
| Scope | Intra-instance (local sessions, containers, recursive sub-nodes) + federation peers |
| Memory isolation | Hard boundary: a peer can ONLY access discussion scopes they are participants in. Enforced at API layer. No enumeration of other scopes. |
| Throttle | Per-participant: 60 ops/min default, configurable. 100ms write-batching window. |
| Contention | Per-discussion write lock (not global). |
| History | WAL already provides this ‚ÄĒ no separate history store needed. |
| Loop prevention | `origin_peer + origin_wal_seq` as globally unique identity. `propagated_by[]` tracks path. Receiver skips if already in WAL. |

### Architecture

**New memory entry fields:**
```
scope_type:      "discussion"          (new; existing types unchanged)
discussion_id:   "<uuid>"              (foreign key to discussion metadata)
origin_peer:     "<peer_name>"         (who wrote it)
origin_wal_seq:  <int>                 (WAL sequence at origin)
propagated_by:   ["peer_A","peer_B"]   (hop path for loop detection)
conflict_state:  "none|pending|resolved"
conflicts_with:  ["<entry_id>", ...]   (populated on conflict detection)
```

**Discussion metadata (new table/collection):**
```
discussion_id:   <uuid>
name:            "string"              (human label)
participants:    ["peer_A", "self", "session:abc", "container:xyz"]
created_by:      "peer_name|self"
created_at:      <timestamp>
sync_cursors:    { "peer_A": 847, "peer_B": 612 }   (WAL seq per peer for delivery)
```

**Sync protocol:**
```
1. Writer commits entry to local WAL ‚Üí discussion entry stored
2. Writer identifies all online participants (exclude self)
3. Writer pushes entry to each participant via POST /api/memory/discussions/{id}/sync
4. Receiver: checks origin_peer + origin_wal_seq in local WAL
   - If already seen: drop (loop prevention)
   - If new: store locally; update sync_cursor; fire on_memory_sync notification
5. Receiver re-pushes to its participants not in propagated_by[] (mesh propagation)
6. Offline participant: entry queued; pushed on reconnect (flush in WAL-seq order)
```

**Conflict detection:**
```
Two entries are conflicting if:
  - same discussion_id
  - both have conflict_state="none"
  - content overlaps by semantic similarity > threshold (LLM-optional) OR
  - same memory key/topic (exact match)
  - created within the contention window (default: 5s)

On detection:
  - Both entries set conflict_state="pending"
  - Both set conflicts_with to each other's ID
  - Notification fires to all participants
```

**Memory isolation enforcement:**
```go
// In every memory handler, before any query:
if peerCtx := fedPeerFromContext(r.Context()); peerCtx != nil {
    // Peer can ONLY access their participated discussions
    allowedScopes := discussionStore.ParticipatedScopes(peerCtx.Name)
    if len(allowedScopes) == 0 {
        // Return empty, not error ‚ÄĒ no enumeration leak
        writeJSON(w, emptyMemoryResponse)
        return
    }
    query = query.WhereScope(allowedScopes...)
}
```
